# 微信小程序电商前端设计

## 背景
当前仓库已经具备订单、支付、库存、网关与后台管理能力，并计划新增独立 `product-service` 作为商品主数据来源。现在需要补充一个面向 C 端用户的微信小程序，用于完成最基础的商品浏览与下单流程。

根据商品服务设计，首版商品主数据仅包含 `product_id`、`product_name`、`original_price`、`status`，并明确不在本次提供图片、描述、分类等扩展属性。订单创建时前端不再传商品名称与价格，而只传 `product_id` 与 `quantity`，由订单服务从商品服务获取权威商品快照并计算金额。

因此，小程序首版应围绕“最小可下单闭环”设计，避免引入购物车、多商品合单、支付、地址簿等尚未具备后端支撑或不在首期范围内的能力。

## 目标
- 提供微信小程序首页商品列表。
- 支持用户从商品列表直接发起单商品购买。
- 提供结算页，并支持微信原生收货地址选择。
- 支持调用后端订单接口创建订单。
- 提供下单成功反馈页。
- 前端设计与 `product-service` 首版能力保持一致。

## 非目标
- 不在首版接入微信支付。
- 不在首版实现购物车。
- 不在首版支持多商品合单。
- 不在首版实现系统内地址管理、地址列表、新增编辑地址。
- 不在首版实现商品详情富文本、商品分类、营销活动、优惠券。
- 不在首版实现订单列表与订单详情页。

## 方案概览
首版微信小程序采用三页式最小交易链路：
1. 首页商品列表页。
2. 结算页。
3. 下单成功页。

用户从首页浏览商品后，点击“立即购买”进入结算页。结算页展示当前商品信息，并调用微信原生收货地址选择器获取地址。提交订单时，前端仅向后端发送 `product_id` 与 `quantity`，以及所选地址信息，不再发送商品名称与价格。订单创建成功后跳转下单成功页。

由于支付能力留在第二阶段，本次只实现“订单创建成功”的前半段交易闭环。

## 页面结构与交互

### 1. 首页商品列表页

#### 页面目标
让用户快速浏览当前可售商品，并直接进入购买流程。

#### 页面模块
- 顶部导航栏：显示小程序标题，例如“商城”。
- 商品列表区域。
- 加载态、空态、错误态。

#### 商品卡片字段
首版商品卡片只依赖商品服务已规划提供的字段：
- `product_id`
- `product_name`
- `original_price`
- `status`

由于商品服务首版不提供图片、描述、分类等字段，首页不应将这些字段作为必需依赖。若视觉上需要更完整展示，可使用前端静态占位图，但该占位图不属于后端接口契约。

#### 展示规则
- 价格以前端将“分”转换为“元”的形式展示。
- 默认只展示可售商品；若接口返回上下架混合数据，则前端过滤 `status = 1` 后再渲染。
- 每个商品卡片展示：
  - 商品名称
  - 商品价格
  - “立即购买”按钮

#### 交互规则
- 页面进入时自动请求商品列表接口。
- 点击“立即购买”后跳转结算页。
- 跳转时传递最小展示信息：
  - `product_id`
  - `product_name`
  - `original_price`
- 如果商品处于不可售状态，则按钮禁用或不展示。

#### 页面状态
- 加载中：显示骨架屏或 loading。
- 空列表：显示“暂无商品”。
- 加载失败：显示错误提示与重试按钮。

### 2. 结算页

#### 页面目标
确认收货地址与购买商品，完成订单提交。

#### 页面模块
1. 收货地址卡片。
2. 商品信息卡片。
3. 金额汇总区域。
4. 底部提交栏。

#### 收货地址卡片
- 默认显示“请选择收货地址”。
- 用户点击后调用微信原生收货地址选择器。
- 选择成功后展示：
  - 收件人姓名
  - 手机号
  - 省、市、区
  - 详细地址

首版不自建地址列表，不维护系统内地址簿，完全依赖微信原生地址能力。

#### 商品信息卡片
- 商品名称
- 商品单价
- 购买数量
- 小计金额

首版购买数量固定为 `1`，不提供加减数量，也不支持多商品合单。

#### 金额汇总区域
- 商品金额
- 运费
- 实付金额

运费规则首版建议固定为 `0`，展示为“免运费”或“包邮”，避免在没有明确后端运费规则时引入额外复杂度。

#### 底部提交栏
- 左侧展示实付金额。
- 右侧按钮为“提交订单”。

#### 数据来源与校验策略
结算页初始化时可先使用首页透传的商品信息完成首屏渲染，但推荐进入页面后根据 `product_id` 再调用一次商品详情接口，用于确认：
- 商品仍然存在。
- 商品仍然处于上架状态。
- 商品价格是否发生变化。

若商品已下架、已删除或接口返回异常，则不允许继续下单，并提示用户返回首页。

#### 交互规则
- 未选择地址时，点击提交订单应拦截并提示“请选择收货地址”。
- 点击提交订单后，按钮进入 loading 并禁用，防止重复提交。
- 调用创建订单接口时，前端只传 `product_id` 与 `quantity`，不传 `product_name` 和 `price`。
- 创建订单成功后跳转下单成功页。
- 创建订单失败后，展示错误提示并恢复提交按钮状态。

### 3. 下单成功页

#### 页面目标
向用户明确反馈订单已创建成功。

#### 页面模块
- 成功状态图标或提示。
- 订单创建成功文案。
- 订单号。
- 商品名称。
- 订单金额。
- 操作按钮：
  - 返回首页
  - 查看订单（预留二期能力）

#### 文案建议
由于支付不在首版范围，下单成功页应强调“订单已创建成功”，而不是“支付成功”。

## 页面流转
1. 用户进入首页，请求商品列表接口。
2. 用户点击某个商品的“立即购买”。
3. 前端跳转至结算页，并带上商品最小展示数据。
4. 结算页可选调用商品详情接口进行二次确认。
5. 用户选择微信原生收货地址。
6. 用户点击“提交订单”。
7. 前端调用订单创建接口。
8. 后端创建订单成功后，前端跳转下单成功页。
9. 用户可返回首页，后续二期再补充订单查询能力。

## 接口对接设计

### 商品列表接口
- `GET /api/v1/products`

前端首版只消费以下字段：
- `product_id`
- `product_name`
- `original_price`
- `status`

若接口支持分页，前端首版可先按单页列表处理；若商品数量未来增多，可再补充分页或下拉加载。

### 商品详情接口
- `GET /api/v1/products/:id`

推荐在结算页使用该接口做商品二次确认。前端至少需要以下字段：
- `product_id`
- `product_name`
- `original_price`
- `status`

### 创建订单接口
订单接口路径以订单服务最终提供的接口为准，但请求体设计需遵守商品服务设计约束：前端只传 `product_id` 与 `quantity`。

建议前端请求体结构为：

```json
{
  "items": [
    {
      "product_id": 1001,
      "quantity": 1
    }
  ],
  "address": {
    "userName": "张三",
    "telNumber": "13800000000",
    "provinceName": "广东省",
    "cityName": "深圳市",
    "countyName": "南山区",
    "detailInfo": "科技园1号楼"
  }
}
```

其中 `address` 字段命名最终以后端订单接口为准，但前端内部建议尽量贴近微信原生地址对象结构，以降低数据转换成本。

## 前端数据模型建议

### Product
```ts
type Product = {
  product_id: number
  product_name: string
  original_price: number
  status: number
}
```

### WechatAddress
```ts
type WechatAddress = {
  userName: string
  telNumber: string
  provinceName: string
  cityName: string
  countyName: string
  detailInfo: string
}
```

### 首页状态
```ts
type ProductListPageState = {
  loading: boolean
  list: Product[]
  error: string
}
```

### 结算页状态
```ts
type CheckoutPageState = {
  product: Product | null
  address: WechatAddress | null
  submitting: boolean
}
```

## 目录结构建议
如果微信小程序作为独立前端项目实现，建议使用以下目录结构：

```txt
miniprogram/
  pages/
    home/
      index.js
      index.wxml
      index.wxss
      index.json
    checkout/
      index.js
      index.wxml
      index.wxss
      index.json
    order-success/
      index.js
      index.wxml
      index.wxss
      index.json
  services/
    product.js
    order.js
  utils/
    request.js
    price.js
    address.js
  app.js
  app.json
  app.wxss
```

这样的结构可以让页面、接口访问、价格格式化和地址处理职责清晰分离，便于首版快速迭代。

## 异常场景与处理
首版至少应覆盖以下异常情况：

1. 商品列表加载失败。
   - 首页展示错误提示与重试按钮。

2. 商品列表为空。
   - 首页展示“暂无商品”。

3. 进入结算页后商品已下架或不存在。
   - 提示商品不可购买，并引导返回首页。

4. 未选择地址即提交订单。
   - 提示用户先选择收货地址。

5. 创建订单失败。
   - 使用 toast 或错误提示反馈，并允许重新提交。

6. 用户重复点击提交。
   - 提交按钮 loading 并禁用，直到请求完成。

7. 商品服务或订单服务短暂不可用。
   - 展示统一错误提示，不在前端做金额兜底或本地下单缓存。

## 测试建议
### 页面与交互验证
- 首页商品列表正常加载。
- 首页空态、错误态渲染正确。
- 点击立即购买能正确进入结算页。
- 微信原生地址选择成功后，页面正确展示地址。
- 未选地址无法提交订单。
- 提交中按钮禁用，防止重复下单。
- 创建订单成功后跳转成功页。
- 创建订单失败后页面可恢复再次提交。

### 联调验证
- 商品改价后，结算页若重新拉商品详情，应展示最新价格。
- 商品下架后，结算页不可继续提交订单。
- 下单请求中不再上传商品名称与价格。
- 订单创建成功后返回的订单号可在成功页展示。

## 二期扩展方向
在不推翻首版结构的前提下，二期可逐步扩展：
- 微信支付能力。
- 订单列表与订单详情页。
- 商品详情页。
- 购物车与多商品合单。
- 商品图片、描述、分类展示。
- 地址簿能力。
- 优惠券、营销活动与运费规则。

## 与现有后端设计的一致性约束
- 小程序首页展示字段必须与商品服务首版输出能力保持一致，不假定后端已有图片与描述字段。
- 结算页下单时不得传商品价格与商品名称，由订单服务按 `product_id` 查询商品服务后补全。
- 小程序首版不涉及支付流程，因此成功页的业务语义是“订单创建成功”而不是“支付成功”。
- 任何商品上下架与改价结果，均以后端查询结果为准，前端只负责展示与提交流程。

## Implementation status
- First-version implementation plan assumes the mini-program is built as a standalone project under `miniprogram/`.
- Product list rendering depends only on `product_id`, `product_name`, `original_price`, and `status`.
- Checkout sends only `product_id`, `quantity`, and address data.
- Payment remains intentionally out of scope.
